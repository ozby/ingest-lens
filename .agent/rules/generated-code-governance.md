# Generated code governance

Generated artifacts are **read-only at the source level**. Edits must flow
from regenerating, not from patching the output.

## Scope

This rule applies to:

- Anything under `.generated/` or `*.generated.ts` / `*.generated.json`.
- OpenAPI / protobuf / GraphQL codegen outputs.
- `pnpm-lock.yaml` (regenerated by `pnpm install`, never hand-edited).
- Workspace `dist/` directories (build output).
- Vendored SDKs under `infra/sdks/` (Pulumi terraform-bridge packages).

## Rules

1. **Never hand-edit generated code.** Regenerate with the declared generator command.
2. **Never import through a package-local mirror of generated code.** Use the canonical import path. Mirror imports create silent drift.
3. **Generators live in one place per kind.** Co-locate the generator command, input, and output directory; document the trio in the owning package's `README.md`.
4. **Regeneration is a one-liner.** Every generator must be runnable as `pnpm --filter <owner> codegen` (or equivalent) with no interactive prompts.
5. **Generated output is gitignored unless it affects the public API.** Ship it when downstream packages need it at install time; derive it at build time otherwise.
6. **Generator inputs are versioned in the repo.** The schema, the spec, or the source-of-truth manifest — whichever drives generation — is a first-class artifact reviewed like code.

## Forbidden

- `// prettier-ignore` on generated files to keep hand edits.
- Deep relative imports into `.generated/` from application code (use the package boundary).
- Editing `pnpm-lock.yaml` by hand, including to resolve merge conflicts. Regenerate with `pnpm install`.

## When generation drift shows up

- CI fails on `pnpm build` because a consumer sees stale types → regenerate in the PR that causes the drift.
- Lockfile has manual conflicts → `pnpm install` and commit the result.
- A generator starts producing different output given the same input → file a blueprint; the generator version is now pinned (or should be).
